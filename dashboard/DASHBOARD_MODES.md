# Режимы работы Dashboard

Dashboard поддерживает два режима работы для классификации обращений:

## 🔵 Demo режим (по умолчанию)

**Путь:** Dashboard → ML Service (POST /classify)

**Характеристики:**
- ⚡ Быстрая классификация (прямой вызов ML Service)
- ❌ **НЕ создает запись в `ticket_events`**
- ❌ Нет полного аудита обращений
- ✅ Подходит для демонстрации и тестирования
- ✅ Не требует запуска Worker в ML Service

**Использование:**
- Установите `DASHBOARD_MODE=demo` (по умолчанию)
- Или выберите "demo" в переключателе на странице Dashboard

## 🔒 Production режим

**Путь:** Dashboard → Ingestion Service → ML Service → Output Service

**Характеристики:**
- ✅ **Создает запись в `ticket_events`** при каждом обращении
- ✅ Полный аудит всех классификаций
- ✅ Полный pipeline обработки (классификация → отправка в Jira/файлы)
- ⏱️ Медленнее (требует обработки через очередь)
- ✅ Подходит для production использования

**Требования:**
- Ingestion Service должен быть запущен (порт 8000)
- ML Service должен быть запущен (порт 8001)
- Worker в ML Service должен быть включен (`WORKER_ENABLED=true`) для автоматической обработки очереди
- Или можно использовать REST API для ручной обработки

**Использование:**
- Установите `DASHBOARD_MODE=production`
- Или выберите "production" в переключателе на странице Dashboard

## Переключение режимов

### Через переменные окружения:

```bash
# Demo режим (по умолчанию)
export DASHBOARD_MODE=demo

# Production режим
export DASHBOARD_MODE=production
```

### Через UI Dashboard:

На странице Demo есть переключатель режима работы в верхней части страницы.

## Критичность проблемы

**Проблема:** В demo режиме классификации из Dashboard не логируются в `ticket_events`.

**Критичность:**
- 🔴 **Высокая для production** - нет полного аудита обращений
- 🟡 **Средняя для разработки** - можно использовать для тестирования
- ✅ **Решено** - добавлен production режим с полным логированием

## Рекомендации

1. **Для разработки и демонстрации:** используйте `demo` режим
2. **Для production:** используйте `production` режим
3. **Для мониторинга:** проверяйте таблицу `ticket_events` в БД для отслеживания всех обращений

## Технические детали

### Demo режим:
```
Dashboard → POST /classify → ML Service → Результат
```

### Production режим:
```
Dashboard → POST /tickets → Ingestion Service
           ↓
           Создание записи в ticket_events
           ↓
           Добавление в очередь Redis
           ↓
           ML Service Worker обрабатывает очередь
           ↓
           Обновление ticket_events с результатами
           ↓
           Отправка в Output Service (Jira/файлы)
           ↓
           Dashboard получает результат через polling
```

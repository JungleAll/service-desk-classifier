Shared Utilities

Назначение
- Общие утилиты для микросервисов: база данных, Redis и логирование

Файлы
- database.py: пул подключений к PostgreSQL, контекстный менеджер курсора
- redis_client.py: клиент Redis, префиксы ключей и TTL для кэша предсказаний
- logger.py: централизованный модуль логирования с поддержкой JSON формата для ElasticSearch
- requirements.txt: зависимости общего уровня

ENV для PostgreSQL
- POSTGRES_HOST, POSTGRES_PORT, POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD

ENV для Redis
- REDIS_HOST, REDIS_PORT, REDIS_PASSWORD (опционально)
- REDIS_DB_QUEUES (по умолчанию: 0) - база данных для очередей
- REDIS_DB_CACHE (по умолчанию: 1) - база данных для кэша
- Ключевые сущности:
  - DB 0 (очереди): pending_tickets, failed_tickets
  - DB 1 (кэш): cache_predictions (TTL 1 час)
- **Архитектура Redis:** Разделение на разные базы данных для изоляции очередей и кэша
  - См. `REDIS_ARCHITECTURE.md` для подробностей

ENV для Логирования
- LOG_LEVEL: уровень логирования (DEBUG, INFO, WARNING, ERROR, CRITICAL, по умолчанию: INFO)
- LOG_DIR: директория для сохранения логов (по умолчанию: ./logs)
- LOG_ENABLE_FILE: включить файловое логирование (по умолчанию: true)
- LOG_ENABLE_JSON: включить JSON формат для файлов (по умолчанию: true)
- LOG_ENABLE_STDOUT: включить вывод в stdout (по умолчанию: true)
- LOG_MAX_BYTES: максимальный размер файла перед ротацией (по умолчанию: 10485760 = 10MB)
- LOG_BACKUP_COUNT: количество резервных файлов (по умолчанию: 5)

Рекомендации
- Импортировать из shared вместо дублирования кода
- Использовать централизованный модуль логирования для всех сервисов
- Следить за единым форматом логирования и обработкой ошибок

## Как работает Redis кэш (простыми словами)

### Что такое кэш?

Представьте, что вы каждый день идете в библиотеку за одной и той же книгой. 
Без кэша: каждый раз вы идете в библиотеку, ищете книгу на полке, берете ее, читаете, возвращаете.
С кэшем: вы один раз взяли книгу и положили ее на свой стол. Теперь, когда она нужна снова, вы просто берете ее со стола - намного быстрее!

**Кэш в Redis** - это быстрая память, где хранятся уже вычисленные результаты. Если тот же запрос приходит повторно, система не тратит время на вычисления, а просто берет готовый результат из кэша.

### Как это работает в нашей системе?

Когда пользователь отправляет текст для классификации:

#### ❌ БЕЗ кэша (медленно):

```
1. Пользователь: "У меня не работает принтер"
   ↓
2. Система: "Хм, нужно классифицировать этот текст..."
   ↓
3. Система: Загружает модель ML (тяжелая операция)
   ↓
4. Система: Обрабатывает текст через модель
   ↓
5. Система: Вычисляет вероятности для всех 17 категорий
   ↓
6. Результат: "Запрос на обслуживание" (уверенность 79%)
   ⏱️ Время: ~40-50 миллисекунд
```

Если тот же текст придет снова:
```
1. Пользователь: "У меня не работает принтер" (тот же текст!)
   ↓
2. Система: "Опять нужно классифицировать..."
   ↓
3. Система: Снова загружает модель, обрабатывает, вычисляет...
   ↓
6. Результат: "Запрос на обслуживание" (тот же результат!)
   ⏱️ Время: ~40-50 миллисекунд (снова!)
```

**Проблема:** Система тратит время на повторные вычисления, хотя результат уже известен!

#### ✅ С кэшем (быстро):

```
1. Пользователь: "У меня не работает принтер"
   ↓
2. Система: "Проверяю кэш..."
   ↓
3. Система: "В кэше нет, классифицирую..."
   ↓
4. Система: Обрабатывает через модель ML
   ↓
5. Результат: "Запрос на обслуживание" (уверенность 79%)
   ↓
6. Система: "Сохраняю результат в кэш на 1 час"
   ⏱️ Время: ~40-50 миллисекунд (первый раз)
```

Если тот же текст придет снова:
```
1. Пользователь: "У меня не работает принтер" (тот же текст!)
   ↓
2. Система: "Проверяю кэш..."
   ↓
3. Система: "О! Есть в кэше! Беру готовый результат"
   ↓
4. Результат: "Запрос на обслуживание" (уверенность 79%)
   ⏱️ Время: ~20-30 миллисекунд (в 2 раза быстрее!)
```

**Преимущество:** Система мгновенно возвращает результат из кэша, не тратя время на вычисления!

### Практический пример

Вот как это выглядит в коде:

```python
from shared.redis_client import get_cache, set_cache, CACHE_PREDICTIONS
import hashlib

# Текст для классификации
text = "У меня не работает принтер"

# Создаем ключ кэша (хэш текста + версия модели)
text_hash = hashlib.md5(text.encode('utf-8')).hexdigest()
cache_key = f"{CACHE_PREDICTIONS}:v1.0:{text_hash}"

# ПРОВЕРКА: Есть ли результат в кэше?
cached_result = get_cache(cache_key)

if cached_result:
    # ✅ КЭШ НАЙДЕН - возвращаем готовый результат
    print("Результат из кэша (быстро!)")
    predicted_type = cached_result['predicted_type']
    confidence = cached_result['confidence']
    print(f"Категория: {predicted_type}, Уверенность: {confidence:.2%}")
else:
    # ❌ КЭША НЕТ - нужно классифицировать
    print("Кэша нет, классифицирую (медленно)...")
    
    # Здесь происходит тяжелая работа:
    # - Загрузка ML модели
    # - Обработка текста
    # - Вычисление вероятностей
    result = classify_text(text)  # Это занимает время!
    
    predicted_type = result['predicted_type']
    confidence = result['confidence']
    
    # СОХРАНЯЕМ результат в кэш на 1 час (3600 секунд)
    set_cache(cache_key, {
        'predicted_type': predicted_type,
        'confidence': confidence,
        'model_version': 'v1.0',
        'decision': result['decision']
    }, ttl=3600)
    
    print(f"Результат сохранен в кэш. Категория: {predicted_type}")
```

### Визуальное сравнение

**Сценарий:** 100 пользователей отправляют одинаковый текст "Не работает принтер"

#### Без кэша:
```
Запрос 1: Классификация... ⏱️ 45ms
Запрос 2: Классификация... ⏱️ 45ms
Запрос 3: Классификация... ⏱️ 45ms
...
Запрос 100: Классификация... ⏱️ 45ms

Общее время: 100 × 45ms = 4500ms (4.5 секунды)
```

#### С кэшем:
```
Запрос 1: Классификация... ⏱️ 45ms → Сохранено в кэш
Запрос 2: Из кэша... ⏱️ 25ms ✅
Запрос 3: Из кэша... ⏱️ 25ms ✅
...
Запрос 100: Из кэша... ⏱️ 25ms ✅

Общее время: 45ms + (99 × 25ms) = 2520ms (2.5 секунды)
Экономия: почти в 2 раза быстрее!
```

### Важные детали

1. **TTL (Time To Live)** - время жизни кэша
   - По умолчанию: 3600 секунд (1 час)
   - После истечения кэш автоматически удаляется
   - Это нужно, чтобы кэш не устаревал при обновлении модели

2. **Ключ кэша** включает версию модели:
   ```
   cache_predictions:v1.0:abc123def456...
   ```
   Если модель обновится до v2.0, старый кэш не будет использоваться

3. **Разделение баз данных:**
   - DB 0: Очереди (временные задачи)
   - DB 1: Кэш (результаты классификации)
   - Это позволяет управлять ими независимо

### Когда кэш особенно полезен?

✅ **Полезен:**
- Повторяющиеся запросы (один и тот же текст)
- Популярные вопросы ("Не работает принтер", "Нужна помощь")
- Пакетная обработка одинаковых текстов

❌ **Не так полезен:**
- Уникальные запросы (каждый раз новый текст)
- Редкие запросы (текст приходит один раз)

### Как проверить работу кэша?

```python
from shared.redis_client import get_cache_info, get_cache

# Информация о кэше
info = get_cache_info()
print(f"Ключей в кэше: {info['keys']}")
print(f"Использовано памяти: {info['used_memory']}")

# Проверка конкретного ключа
cache_key = "cache_predictions:v1.0:abc123..."
result = get_cache(cache_key)
if result:
    print("Кэш найден!")
    print(f"Результат: {result['predicted_type']}")
else:
    print("Кэш не найден")
```

## Централизованное логирование (logger.py)

### Назначение

Модуль `logger.py` предоставляет единый интерфейс для логирования во всех микросервисах с поддержкой:
- **JSON формата** для интеграции с ElasticSearch
- **Файлового вывода** с автоматической ротацией
- **Stdout вывода** для совместимости с Docker logs
- **Структурированных логов** с метаданными

### Использование

```python
from shared.logger import configure_service_logging

# Настройка логирования для сервиса
logger = configure_service_logging("ingestion")

# Базовое логирование
logger.info("Сообщение")
logger.warning("Предупреждение")
logger.error("Ошибка", exc_info=True)

# Логирование с дополнительными полями
logger.info(
    "Обращение обработано",
    extra={"ticket_id": "tick_12345678", "duration_ms": 150}
)
```

### Форматы логов

#### JSON формат (для файлов)
```json
{
  "@timestamp": "2025-01-19T10:30:45.123Z",
  "service": "ingestion",
  "level": "INFO",
  "logger": "ingestion",
  "message": "Обращение tick_12345678 создано",
  "module": "app",
  "function": "create_ticket",
  "line": 207,
  "ticket_id": "tick_12345678"
}
```

#### Стандартный формат (для stdout)
```
2025-01-19 10:30:45 - ingestion - INFO - Обращение tick_12345678 создано
```

### Расположение логов

- **В Docker контейнерах:** `/app/logs/{service}.log`
- **Локально:** `./logs/{service}.log`

Файлы логов:
- `ingestion.log` - логи Ingestion Service
- `ml.log` - логи ML Service (API)
- `ml.worker.log` - логи ML Worker
- `config.log` - логи Config Service
- `output.log` - логи Output Service

### Ротация логов

Логи автоматически ротируются при достижении максимального размера:
- `service.log` - текущий файл
- `service.log.1` - предыдущий файл
- `service.log.2` - файл на 2 ротации назад
- ... и так далее до `service.log.5`

### Интеграция с ElasticSearch

JSON формат логов готов для интеграции через:
- **Filebeat** → Logstash → ElasticSearch
- **Fluentd** → ElasticSearch
- Прямая отправка через Python библиотеку `elasticsearch`

Подробная документация по интеграции: `LOGGING_GUIDE.md` (в корне проекта)

### Примеры использования

```python
# Простое логирование
logger.info("Сервис запущен")

# Логирование с контекстом
logger.info(
    "Тикет обработан",
    extra={
        "ticket_id": "tick_123",
        "request_id": "req_abc",
        "duration_ms": 150
    }
)

# Логирование ошибок
try:
    # код
except Exception as e:
    logger.error("Ошибка при обработке", exc_info=True, extra={"ticket_id": "tick_123"})
```

### Дополнительные поля

Модуль поддерживает автоматическое добавление дополнительных полей в JSON логи:
- `ticket_id` - ID тикета
- `request_id` - ID запроса
- `user_id` - ID пользователя
- `duration_ms` - длительность операции в миллисекундах

Эти поля передаются через параметр `extra` в методах логирования.

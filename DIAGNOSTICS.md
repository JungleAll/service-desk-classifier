# Диагностика проблем с обработкой тикетов

## Проблема: Тикет в статусе "queued", но не обрабатывается

### Шаг 1: Проверка статуса Worker

Вызовите диагностический endpoint:

```bash
curl http://localhost:8001/worker/diagnostics
```

Ответ покажет:
- `worker_enabled`: включен ли Worker
- `worker_running`: запущен ли Worker
- `model_loaded`: загружена ли модель
- `queue_pending_length`: количество тикетов в очереди
- `redis_connected`: подключен ли Redis

### Шаг 2: Если Worker не включен

**Проблема:** `worker_enabled: false`

**Решение:** Установите переменную окружения и перезапустите ML Service:

```bash
export WORKER_ENABLED=true
# Перезапустите ML Service
```

Или в Docker Compose добавьте:
```yaml
ml-service:
  environment:
    - WORKER_ENABLED=true
```

### Шаг 3: Если Worker включен, но не запущен

**Проблема:** `worker_enabled: true`, но `worker_running: false`

**Решение:** Проверьте логи ML Service на наличие ошибок при запуске Worker.

### Шаг 4: Если модель не загружена

**Проблема:** `model_loaded: false`

**Решение:** 
1. Проверьте наличие файлов модели в `models/v1.0/`:
   - `classifier_smote_new.pkl`
   - `vectorizer_smote.pkl`
   - `label_encoder_smote.pkl`

2. После добавления модели:
   - Вызовите `POST http://localhost:8001/reload_model` для немедленной перезагрузки
   - Или перезапустите ML Service
   - Worker автоматически попытается перезагрузить модель при обработке следующего тикета

### Шаг 5: Если Redis недоступен

**Проблема:** `redis_connected: false`

**Решение:**
1. Проверьте, запущен ли Redis:
   ```bash
   redis-cli ping
   # Должно вернуть: PONG
   ```

2. Проверьте переменные окружения:
   - `REDIS_HOST` (по умолчанию: localhost)
   - `REDIS_PORT` (по умолчанию: 6379)
   - `REDIS_PASSWORD` (если используется)

### Шаг 6: Если тикеты в очереди, но не обрабатываются

**Проблема:** `queue_pending_length > 0`, но Worker не обрабатывает

**Решение:**
1. Проверьте логи ML Service - должны быть сообщения:
   ```
   Worker: Получен тикет {ticket_id} из очереди, начинаю обработку...
   ```

2. Если таких сообщений нет:
   - Проверьте, что Worker действительно запущен (`worker_running: true`)
   - Проверьте подключение к Redis
   - Проверьте, что модель загружена

3. Проверьте логи на наличие ошибок:
   ```bash
   # В логах ML Service должны быть сообщения о получении тикетов
   ```

### Шаг 7: Проверка через health endpoint

```bash
curl http://localhost:8001/health
```

Ответ должен содержать:
- `status: "healthy"`
- `model_loaded: true`
- `message: "Сервис работает нормально (Worker: running)"` (если Worker включен)

### Шаг 8: Ручная проверка очереди Redis

```bash
# Подключитесь к Redis
redis-cli

# Переключитесь на DB 0 (очереди)
SELECT 0

# Проверьте длину очереди
LLEN pending_tickets

# Посмотрите первый тикет (без удаления)
LRANGE pending_tickets 0 0
```

### Шаг 9: Переобработка failed тикета

Если тикет в статусе `failed`, переотправьте его:

```bash
curl -X POST "http://localhost:8000/tickets/{ticket_id}/reprocess"
```

После этого проверьте:
1. Тикет должен быть в статусе `queued`
2. Worker должен получить его из очереди
3. Тикет должен быть обработан

## Типичные проблемы и решения

### Проблема 1: Worker не запускается

**Причина:** `WORKER_ENABLED=false` (по умолчанию)

**Решение:** Установите `WORKER_ENABLED=true` и перезапустите ML Service

### Проблема 2: Модель не загружена

**Причина:** Файлы модели отсутствуют в `models/v1.0/`

**Решение:** 
1. Добавьте файлы модели
2. Вызовите `POST /reload_model` или перезапустите сервис
3. Worker автоматически попытается перезагрузить модель при обработке тикета

### Проблема 3: Redis недоступен

**Причина:** Redis не запущен или неправильные настройки подключения

**Решение:** 
1. Запустите Redis
2. Проверьте переменные окружения `REDIS_HOST`, `REDIS_PORT`

### Проблема 4: Тикет в очереди, но не обрабатывается

**Причина:** Worker не может получить тикет из очереди или модель не загружена

**Решение:**
1. Проверьте статус Worker через `/worker/diagnostics`
2. Проверьте логи ML Service
3. Убедитесь, что модель загружена

## Быстрая диагностика

Выполните все проверки одной командой:

```bash
# 1. Проверка Worker и очереди
curl http://localhost:8001/worker/diagnostics | jq

# 2. Проверка здоровья сервиса
curl http://localhost:8001/health | jq

# 3. Проверка статуса модели
curl http://localhost:8001/model/status | jq
```

## Логи для отладки

Включите DEBUG логирование:

```bash
export LOG_LEVEL=DEBUG
# Перезапустите ML Service
```

В логах вы увидите:
- Попытки получения тикетов из очереди
- Обработку каждого тикета
- Ошибки при загрузке модели
- Проблемы с подключением к Redis

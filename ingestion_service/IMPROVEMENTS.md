# Улучшения Ingestion Service

## Дата обновления: 2025-01-19

## 1. Улучшенные сообщения об ошибках

### Проблема
Ранее сообщения об ошибках были техническими и неинформативными для конечных пользователей.

### Решение
Все сообщения об ошибках теперь:
- **Понятны для пользователей** - используют простой язык без технических деталей
- **Содержат рекомендации** - указывают, что делать (повторить попытку, обратиться к администратору)
- **Разделены по типам ошибок:**
  - **503 Service Unavailable** - для временных проблем (недоступность БД, Redis)
  - **500 Internal Server Error** - для внутренних ошибок с ID тикета для отслеживания

### Примеры улучшенных сообщений

**До:**
```
Ошибка при создании обращения: connection to server at "localhost" (::1), port 5432 failed
```

**После:**
```
База данных PostgreSQL недоступна. Пожалуйста, повторите попытку позже.
```

**До:**
```
Ошибка при получении списка: {str(e)}
```

**После:**
```
Внутренняя ошибка при получении списка обращений. Обратитесь к администратору системы.
```

### Обработанные типы ошибок

1. **PostgreSQL недоступен** (`psycopg2.OperationalError`, `psycopg2.InterfaceError`)
   - Сообщение: "База данных PostgreSQL недоступна. Пожалуйста, повторите попытку позже."
   - HTTP статус: 503 Service Unavailable

2. **Redis недоступен**
   - Сообщение: "Не удалось добавить обращение в очередь обработки. Redis может быть недоступен."
   - Тикет помечается как `failed` в БД

3. **Внутренние ошибки**
   - Сообщение: "Внутренняя ошибка при создании обращения. Обратитесь к администратору системы. (ID: {ticket_id})"
   - HTTP статус: 500 Internal Server Error
   - Включает ID тикета для отслеживания

## 2. Метрики недоступности соседних сервисов

### Описание
Добавлена система отслеживания метрик недоступности соседних сервисов (Config Service, PostgreSQL, Redis).

### Функциональность

#### Модуль метрик (`ingestion_service/metrics.py`)
- **Счетчики недоступности** - общее количество событий недоступности для каждого сервиса
- **Время последней недоступности** - когда последний раз был недоступен сервис
- **История событий** - последние 100 событий недоступности для каждого сервиса
- **Время последней успешной проверки** - когда последний раз успешно подключились к сервису
- **Автоматическое логирование** - все события недоступности логируются

#### Endpoint метрик

**GET `/metrics/unavailability`**

Возвращает метрики недоступности для всех соседних сервисов:

#### Обновленный Healthcheck

**GET `/health`**

Теперь включает статус Config Service:

```json
{
  "status": "healthy",
  "redis": "connected",
  "postgresql": "connected",
  "config_service": "connected"
}
```

### Автоматическое отслеживание

Метрики автоматически обновляются при:
- **Успешных операциях** - вызывается `record_success(service)`
- **Ошибках подключения** - вызывается `record_unavailability(service, error_message)`

### Использование метрик

1. **Мониторинг** - отслеживание доступности соседних сервисов
2. **Алертинг** - настройка алертов при длительной недоступности
3. **Анализ** - анализ истории недоступности для выявления проблемных периодов
4. **Отладка** - просмотр последних событий для диагностики проблем

### Примеры использования

#### Проверка метрик через API

```bash
curl http://localhost:8000/metrics/unavailability
```

#### Проверка статуса конкретного сервиса

```python
from ingestion_service.metrics import get_service_status

status = get_service_status("postgresql")
if not status["is_available"]:
    print(f"PostgreSQL недоступен с {status['last_unavailable_at']}")
```

#### Настройка алертов

```python
from ingestion_service.metrics import get_metrics

metrics = get_metrics()
for service, data in metrics.items():
    if not data["is_available"]:
        duration = data["time_since_unavailable_seconds"]
        if duration > 300:  # 5 минут
            send_alert(f"Сервис {service} недоступен более 5 минут")
```

## 3. Улучшенная обработка ошибок

### Config Service
- Добавлена обработка таймаутов и ошибок подключения
- Автоматическое отслеживание недоступности
- Fallback на включенное состояние при недоступности

### PostgreSQL
- Специфичная обработка `psycopg2.OperationalError` и `psycopg2.InterfaceError`
- Понятные сообщения для пользователей
- Автоматическое отслеживание недоступности

### Redis
- Отслеживание успешных и неудачных операций
- Автоматическое обновление метрик
- Логирование всех событий недоступности

## 4. Потокобезопасность

Модуль метрик использует `threading.Lock` для обеспечения потокобезопасности при одновременном доступе из разных запросов.

## 5. Производительность

- История событий ограничена 100 последними событиями (используется `deque` с `maxlen=100`)
- Минимальные накладные расходы на отслеживание метрик
- Метрики хранятся в памяти (не требуют БД)

## Переменные окружения

Нет дополнительных переменных окружения - все работает из коробки.

## Миграция

Изменения обратно совместимы:
- Все существующие эндпоинты работают как раньше
- Новые эндпоинты добавлены без изменения существующих
- Сообщения об ошибках улучшены, но структура ответов не изменилась

## Тестирование

Для тестирования метрик можно использовать функцию сброса:
